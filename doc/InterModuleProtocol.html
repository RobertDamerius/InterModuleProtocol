Overview

The intermodule protocol is designed to exchange basic messages between modules of a GNC system. Only generic messages are defined that are independent of the application and therefore independent of the vehicle type. All further vehicle-specific messages must still be developed additionally.

Basic Message Structure

A message consists of an ID, a timestamp and the actual data that depends on the ID. The basic protocol structure is as follows.

ID
(1 byte)		 timestamp
(5 bytes)		 data
(N bytes)

ID

The ID indicates the type of the message.

timestamp

The timestamp indicates the validity time of the message. It's given as the UTC second of the day with a resolution of 100 nanoseconds. A time of 7 o'clock (UTC) therefore results in a value of 7*3600*10000000=252000000000 (0x3AAC5ED800).

data

The actual data of variable length that depends on the ID.

Endianess

The network byte order is used which is defined to always be big-endian (MSB first).

(0x00) Info Request Message

The info request message is used to request information from a module. A module belongs to a vehicle. A module that receives this message should respond with an info response message.

ID (0x00)
(1 byte)		 timestamp
(5 bytes)

ID

The ID is set to 0x00.

timestamp

The timestamp, given in UTC seconds of the day, that indicates the validity time of the message. It's an unsigned value (5 bytes) with a resolution of 100 nanoseconds.

(0x01) Info Response Message

The info response message is the response to an info request message. A module belongs to a vehicle and sends this message to indicate its belonging to the vehicle.

ID (0x01)
(1 byte)		 timestamp
(5 bytes)		 dimension
(12 bytes)		 offset
(12 bytes)		 vehicleName
(N bytes)		 zero-terminator
(1 byte)		 description
(N bytes)		 zero-terminator
(1 byte)

ID

The ID is set to 0x01.

timestamp

The timestamp, given in UTC seconds of the day, that indicates the validity time of the message. It's an unsigned value (5 bytes) with a resolution of 100 nanoseconds.

dimension

The dimension of the default vehicle cuboid in meters (body-frame). The datatype is float and the format is as follows.

x dimension
(4 bytes)		 y dimension
(4 bytes)		 z dimension
(4 bytes)

offset

The offset of the default vehicle cuboid in meters (body-frame). It's the position of the cuboid with respect to the origin of the body-frame. The datatype is float and the format is as follows.

x offset
(4 bytes)		 y offset
(4 bytes)		 z offset
(4 bytes)

vehicleName

The text string (UTF-8) that indicates the vehicle name. Make sure that there is no null character ('\0') in the text.

zero-terminator

The zero-terminator denotes the end of the vehicleName string and is always 0x00.

description

The text string (UTF-8) that indicates an optional description string. Make sure that there is no null character ('\0') in the text.

zero-terminator

The zero-terminator denotes the end of the description string and is always 0x00.

guidance state

If an availability bit is set to zero (not available) then the corresponding state values won't be added. (0x4C) Log Message

The log message is used to provide text logs. The message structure is as follows.

ID (0x4C)
(1 byte)		 timestamp
(5 bytes)		 color
(3 bytes)		 text
(N bytes)		 zero-terminator
(1 byte)

Maximum UDP message length

A UDP message can contain at most 65507 bytes. The maximum number of bytes in the text is then limited to 65497.

ID

The ID is set to 0x4C which is the ASCII-code for 'L'.

timestamp

The timestamp, given in UTC seconds of the day, that indicates the validity time of the message. It's an unsigned value (5 bytes) with a resolution of 100 nanoseconds.

color

The color (RGB) that should be used to display the text. The format is as follows.

red
(1 byte)		 green
(1 byte)		 blue
(1 byte)

text

The text string (UTF-8). Make sure that there is no null character ('\0') in the text.

zero-terminator

The zero-terminator denotes the end of the text string and is always 0x00.

(0x50) Convex Polygon Message

A convex polygon message is used to transfer a variable number of convex polygons. A convex polygon must constist of at least three vertices. An edge is indicated by two following vertices. Multiple vertices must not lie on one line, e.g. the angle of two subsequent edges must not be the same. There must be no duplicated vertices. There are static and dynamic polygons. All velocities of a dynamic polygon refer to the mean value of all vertices of this polygon. Each vertex has an x and a y coordinate, which is given in meters. The x-axis points to north while the y-axis points to east. The polygon message contains a geographic position that denotes the origin to which the vertex coordinates refer.

Maximum UDP message length

A UDP message can contain at most 65507 bytes. It may be necessary to split a polygon dataset into several messages. For this purpose there is a segment counter in the polygon message.

ID

The ID is set to 0x50 which is the ASCII-code for 'P'.

timestamp

The timestamp, given in UTC seconds of the day, that indicates the validity time of the message. It's an unsigned value (5 bytes) with a resolution of 100 nanoseconds.

maxSegmentIndex

The maximum zero-based segment index that indicates the last message segment. That is, the maximum number of polgon messages belonging to a polygon dataset is (maxSegmentIndex + 1).

segmentIndex

The current zero-based segment index for this message.

originLLA

Geographic position (latitude, longitude, altitude) used as origin for all polygons of the message. Note, that the network byte order is big-endian.

longitudedoubleGeographic position (latitude in radians) according to WGS84, range [-pi, +pi).
altitudedoubleGeographic position (altitude in meters) according to WGS84, positive upwards.

numPolygons

A 16-bit unsigned value that indicates the number of polygons that are contained in the polygon data field. Note, that the network byte order is big-endian.

polygon data

The number of polygons is indicated by numPolygons. All polygons are specified one after the other. The data for a polygon contains the following data.

polygonID

A unique 16-bit number that identifies the polygon. The datatype is uint16.

classification

An 8-bit value that indicates a classification of the polygon. The most significant bit indicates whether the polygon is a static or a dynamic polygon. All values in range [0, 127] denote static polygons and all values in range [128,255] denote dynamic polygons. The actual classifications in the corresponding ranges are not specified and can be defined depending on the application. Note that the following velocities are only sent for dynamic polygons.

velocities

The velocities are only available if the polygon is a dynamic polygon, that is if the most significant bit of the classification is set. All velocities of a polygon refer to the mean value of all vertices of this polygon. The format is as follows.

NameData typeDescription
velocityNorthfloatVelocity in north direction in meters per second.
velocityEastfloatVelocity in east direction in meters per second.
yawRatefloatRotational velocity around downward axis of the navigation frame in radians per second.

numVertices

The number of vertices that are contained in the convex polygon. The datatype is uint8.

vertex data

The number of vertices is indicated by numVertices. All vertices are specified one after the other. The verticies are given in mathematically positive order, that is around the downward axis. The data for a vertex is as follows.

x coordinate
(4 bytes)		 y coordinate
(4 bytes)

x coordinate

The x coordinate (north-direction) of the vertex in meters with respect to the geographic origin (originLLA). The datatype is float.

y coordinate

The y coordinate (east-direction) of the vertex in meters with respect to the geographic origin (originLLA). The datatype is float.

(0x54) Guidance Trajectory Message

A guidance trajectory message is used to transfer a reference state trajectory that contains a variable number of state points. The message contains a time (UTC) that specifies the start time of the trajectory. A time difference indicates the time interval between two state points. If this time difference is set to zero, then this message corresponds to a guidance waypoint message, where each state point then corresponds to one waypoint. The message structure is as follows.

ID (0x54)
(1 byte)		 timestamp
(5 bytes)		 start time
(8 bytes)		 delta time
(4 bytes)		 numPoints
(2 bytes)		 configuration
(4 bytes)
point data
(N bytes)
Point 1
(State vector according to configuration)		 ... Point P
(State vector according to configuration)

ID

The ID is set to 0x54 which is the ASCII-code for 'T'.

timestamp

The timestamp, given in UTC seconds of the day, that indicates the validity time of the message. It's an unsigned value (5 bytes) with a resolution of 100 nanoseconds.

start time

The start time indicates the beginning of the trajectory. This corresponds to the validity time of the first state point. The time is specified as a UTC date and a UTC timestamp and has the following structure.

UTC date (year, month, day)
(3 bytes)		 UTC timestamp
(5 bytes)

The UTC timestamp is a 5-byte value that has the same format as the timestamp of the message.

delta time

The time difference in seconds between two state points of the reference trajectory. If this time difference is set to zero, then this message corresponds to a guidance waypoint message, where each state point then corresponds to one waypoint. The datatype is float. Note, that the network byte order is big-endian.

numPoints

The number of points that are contained in the trajectory. The datatype is uint16.

point data

The number of points is indicated by numPoints. All points are specified one after the other. The data of a point depends on the configuration bitfield. All values of a state are added one after the other in the same order in which they were configured with the bitfield. If an availability bit is set to zero (not available) then the corresponding state values won't be added. The corresponding representation bit indicates whether a value has a size of 4 or 8 bytes. double is always used for an 8-byte representation. A 4-byte representation uses either a float for infinite state values or an int32 for finite state values to increase the resolution. The following table shows all state values in the correct order. Note, that the network byte order is big-endian!
